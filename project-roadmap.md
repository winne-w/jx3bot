# Project Roadmap

更新时间：2026-03-09

本文记录项目中期方向，不展开具体执行步骤。具体落地任务见 `docs/tasks/`，阶段性方案见 `docs/exec-plans/`。

## 当前阶段

项目正处于“历史插件式实现”向“可维护分层架构”过渡的中段，目标不是一次性重写，而是在保证现有机器人可运行的前提下持续收口。

## 路线图

### 1. 稳定分层边界

目标：

- 固定 `plugins / services / infra / storage / renderers` 的依赖方向
- 避免新功能继续回流到单入口或工具大杂烩

成功标志：

- 新增需求默认按层落位
- `singletons`、`defget` 不再继续膨胀

### 2. 统一运行与观测口径

目标：

- 逐步移除遗留 `print()`
- 统一日志上下文
- 让排障路径可被 runbook 和 agent 复用

成功标志：

- 主要业务链路日志都能带命令、服务器、接口上下文
- 定时任务、缓存和外部请求的失败点可快速定位

### 3. 收敛缓存与持久化边界

目标：

- 降低直接文件读写分散在各模块中的情况
- 明确哪些数据属于运行缓存，哪些属于业务持久化

成功标志：

- 高频缓存读写具备更清晰的 repo/adapter 边界
- 当前存储方式、缓存边界和运行数据路径有清晰文档

### 4. 补最小自动化验证

目标：

- 在不引入过重框架的前提下，为高风险链路建立最小自动检查

成功标志：

- 插件加载、关键服务装配、核心接口路径有可重复验证方式
- 手工回归清单与自动化最小集可以互补

## 当前优先级

### P1

- 日志口径收口
- `defget` 兼容导入面缩减
- `status_monitor` 继续收敛边界

### P2

- 缓存访问适配层化
- 高价值命令链路的结构化测试

### P3

- 继续把历史巨型模块拆成可维护单元
- 视后续需要再评估持久化能力扩展方向
